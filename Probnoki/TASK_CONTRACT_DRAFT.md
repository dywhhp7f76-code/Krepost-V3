# TaskContract — черновик контракта задачи для dev-процесса

> Статус: **черновик в Probnoki/**. Ничего в основных файлах не трогает.
> Цель — формализовать передачу задачи между dev-агентами (builder → аудиторы),
> а НЕ рантайм Крепости (Layer 1-4 остаются как есть).

---

## Зачем это

Проблема не в конкретном файле, а в **процессе**:

1. **Деградация в цепочке** — builder и 4 аудитора над одним артефактом;
   свободный текст между ними = двусмысленность = «объяснять по тысяче раз».
2. **Коллюзивная валидация** — если аудиторы однотипны (похожая модель /
   похожий промпт), они одобряют то, что сами бы сгенерировали, и системно
   пропускают один класс багов (hash-divergence, dead-code — проходили ревью).

Лечение: **structured, schema-validated контракт** на точке передачи +
**разнородные аудиторы по механизму проверки, а не по модели**.

---

## Принцип: аудиторы разнородны по МЕХАНИЗМУ

Не «4 копии LLM-ревьюера с разными промптами». Каждый аудитор проверяет
своим инструментом, и большинство проверок — **детерминированные, не LLM**:

| Аудитор | Механизм | Ловит | LLM? |
|---------|----------|-------|------|
| **A. Determinism** | запуск пробника-приёмки, сверка хешей/выводов | hash-divergence, недетерминизм, падение тестов | нет |
| **B. Structure** | AST + import-graph + grep | dead code, дубликаты, битый build-backend, недостижимые ветки | нет |
| **C. Contract** | валидация схемы вывода builder | неполный результат, нарушение API-контракта, тип возврата | нет |
| **D. Logic/Security** | сфокусированное LLM-суждение по узкому чек-листу | fail-closed инварианты, семантика, граничные случаи | да |

Только **D** использует LLM-суждение — и то по узкому чек-листу, а не «глянь,
норм ли». Коллюзия невозможна между A/B/C (это код, не мнение), а D один,
и его вывод перепроверяется исполняемым пробником из A.

---

## Схема контракта

Задача НЕ уходит билдеру без заполненных полей. Билдер НЕ сдаёт результат
без `acceptance` и `deliverable`. Аудитор НЕ выносит вердикт без запуска
своего механизма.

```
TaskContract
├── id                 : str            # "T-042"
├── goal               : str            # одно предложение, что должно стать правдой
├── scope              : ScopeGuard     # какие файлы можно/нельзя трогать
│   ├── allow          : list[glob]     # ["Probnoki/**"]
│   └── forbid         : list[glob]     # ["krepost/security/pipeline.py"] если запрет
├── acceptance         : Acceptance     # КАК проверить, что готово (пишется ДО builder)
│   ├── probnik_path   : str            # исполняемый тест-приёмка (VERIMAP: код)
│   └── criterion_nl   : str            # словесный критерий (VERIMAP: естеств. язык)
├── deliverable        : Deliverable    # что builder ОБЯЗАН вернуть
│   ├── files_changed  : list[str]
│   ├── api_unchanged  : bool           # сломан ли публичный контракт
│   └── summary        : str
└── audit              : list[AuditVerdict]  # по одному от A/B/C/D
    └── AuditVerdict
        ├── auditor    : "A"|"B"|"C"|"D"
        ├── mechanism  : str            # чем именно проверял
        ├── passed     : bool
        └── evidence   : str            # вывод grep/теста/diff — НЕ «мне кажется»
```

**Ключевое правило VERIMAP:** `acceptance.probnik_path` и `criterion_nl`
пишутся **до** того, как builder начал. Критерий приёмки не изобретается
постфактум в голове каждого аудитора — он штампуется вместе с задачей.

## Жизненный цикл

```
1. Оператор пишет TaskContract: goal + scope + acceptance (пробник ещё красный)
2. builder → заполняет deliverable, делает пробник зелёным
3. Аудитор A → запускает пробник, сверяет хеши        → evidence = вывод pytest
4. Аудитор B → AST/grep на dead code, дубли           → evidence = вывод grep
5. Аудитор C → валидирует deliverable против схемы     → evidence = diff схемы
6. Аудитор D → чек-лист fail-closed/логики (LLM)       → evidence = пункты чек-листа
7. Все 4 passed=True И пробник зелёный → задача принята
   Иначе → возврат builder с конкретным evidence (не «переделай»)
```

Возврат билдеру всегда несёт `evidence` — конкретный вывод инструмента,
а не субъективное «не так». Это разрывает цикл «объяснять по тысяче раз».

---

## Что это НЕ

- Не рантайм-компонент Крепости. Layer 1-4 не меняются.
- Не sentinel-агент между builder и аудиторами (оверинжиниринг —
  ручная курация оператором здесь правильнее: проблема в качестве кода,
  не в prompt injection между доверенными dev-агентами).
- Не замена оператору — оператор остаётся финальным гейтом (governance).
